---
title: "REST API: Roles and menu permission endpoints"
description: "Endpoints for managing roles, assigning menus, controlling API-level permissions, and fetching the menu tree used for role-based access control in go-admin."
---

go-admin uses a two-layer permission model. **Menu permissions** control which navigation items a role can see in the front-end sidebar. **API permissions** (stored on the `SysMenu` record via associated `SysApi` entries) gate which back-end routes can be called. Role endpoints manage the roles themselves and their menu assignments; menu endpoints manage the menu tree that defines both the UI structure and the API access rules.

All endpoints require `Authorization: Bearer <token>`. Write operations require a role with the corresponding management permission.

**Base URL:** `http://localhost:8000`

## Menu endpoints

Menus are tree-structured records that define both the front-end sidebar navigation and API-level permissions. Each menu can have associated `SysApi` entries that Casbin uses to allow or deny HTTP requests for a role.

<Note>
  **Menu permissions vs API permissions:** A menu with `menuType = "M"` (directory) or `"C"` (page) controls sidebar visibility. A menu with `menuType = "F"` (button/API rule) controls whether the role may call a specific back-end API route. Both types are assigned to roles through `menuIds` on the role object.
</Note>

### GET /api/v1/menu

List all menu records. The result is not paginated — all matching records are returned. Supports filtering by title and visibility.

#### Query parameters

<ParamField query="title" type="string">
  Partial-match filter on the menu display title.
</ParamField>

<ParamField query="visible" type="number">
  Filter by visibility (`1` = visible, `0` = hidden).
</ParamField>

#### Response fields

<ResponseField name="data" type="object[]">
  <Expandable title="Menu object fields">
    <ResponseField name="menuId" type="number">Unique menu ID.</ResponseField>
    <ResponseField name="menuName" type="string">Internal menu name (component identifier).</ResponseField>
    <ResponseField name="title" type="string">Displayed label in the sidebar.</ResponseField>
    <ResponseField name="icon" type="string">Icon identifier for the menu item.</ResponseField>
    <ResponseField name="path" type="string">Front-end route path.</ResponseField>
    <ResponseField name="paths" type="string">Slash-separated path of parent IDs (breadcrumb trail).</ResponseField>
    <ResponseField name="menuType" type="string">`"M"` = directory, `"C"` = page/component, `"F"` = button or API rule.</ResponseField>
    <ResponseField name="action" type="string">HTTP method for API-type menus (e.g., `"GET"`, `"POST"`).</ResponseField>
    <ResponseField name="permission" type="string">Casbin permission code (e.g., `"system:user:list"`).</ResponseField>
    <ResponseField name="parentId" type="number">ID of the parent menu. `0` = root.</ResponseField>
    <ResponseField name="component" type="string">Front-end Vue component path.</ResponseField>
    <ResponseField name="sort" type="number">Display sort order.</ResponseField>

    <ResponseField name="noCache" type="boolean">Whether to disable front-end component caching.</ResponseField>
    <ResponseField name="breadcrumb" type="string">Whether to include in breadcrumb navigation.</ResponseField>
    <ResponseField name="children" type="object[]">Nested child menu objects.</ResponseField>
    <ResponseField name="sysApi" type="object[]">Associated API rules for permission enforcement.</ResponseField>
  </Expandable>
</ResponseField>

## API registry (sys-api)

The API registry is a catalogue of all registered HTTP routes in go-admin. Each record links an HTTP method + path to a human-readable title and an optional handler name. Menu entries reference API registry records to enforce path-level access control via Casbin. Administrators can view and update registry entries but cannot create them manually — records are seeded by the `migrate` command and updated when new routes are registered.

### GET /api/v1/sys-api

Return a paginated list of registered API entries.

**Query parameters**

<ParamField query="title" type="string">
  Partial-match filter on the API title.
</ParamField>

<ParamField query="path" type="string">
  Partial-match filter on the API path.
</ParamField>

<ParamField query="action" type="string">
  Exact filter on the HTTP method (e.g. `GET`, `POST`, `PUT`, `DELETE`).
</ParamField>

<ParamField query="pageSize" type="integer" default="10">
  Records per page.
</ParamField>

<ParamField query="pageIndex" type="integer" default="1">
  Page number (1-indexed).
</ParamField>

<ResponseField name="data" type="object[]">
  <Expandable title="SysApi object fields">
    <ResponseField name="id" type="integer">Unique API record ID.</ResponseField>
    <ResponseField name="handle" type="string">Handler function name, e.g. `apis.SysUser.GetPage`.</ResponseField>
    <ResponseField name="title" type="string">Human-readable description of the endpoint.</ResponseField>
    <ResponseField name="path" type="string">URL path pattern, e.g. `/api/v1/sys-user`.</ResponseField>
    <ResponseField name="action" type="string">HTTP method, e.g. `GET`.</ResponseField>
    <ResponseField name="type" type="string">Category or grouping label.</ResponseField>
  </Expandable>
</ResponseField>
