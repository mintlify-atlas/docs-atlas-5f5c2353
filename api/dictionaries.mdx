---
title: "REST API: Dictionary data and system config"
description: "Manage dictionary types, dictionary data entries, and system config params in go-admin. Drives dropdown options and app-wide settings via key/value lookup."
---

go-admin uses a two-level dictionary system to store the label/value pairs that power dropdown menus and select inputs throughout the application. A **dictionary type** (e.g., `sys_user_sex`) is the category; **dictionary data** entries (e.g., `"Male" → "1"`, `"Female" → "2"`) are the individual options that belong to that category. Business pages query dictionary data by `dictType` key to populate their select fields at runtime.

The **system config** resource stores key/value application settings. Some config entries are exposed to the frontend (those with `isFrontend = "Y"`); others are back-end-only.

All endpoints require `Authorization: Bearer <token>`. Write operations require a role with management permission.

**Base URL:** `http://localhost:8000`

---

## Dictionary type endpoints

Dictionary types define the named categories for option sets. Each type has a unique `dictType` string that dictionary data entries reference.

### GET /api/v1/dict/type

Return a paginated list of dictionary types.

#### Query parameters

<ParamField query="dictName" type="string">
  Case-insensitive partial-match filter on the dictionary name.
</ParamField>

<ParamField query="dictType" type="string">
  Case-insensitive partial-match filter on the dictionary type key.
</ParamField>

<ParamField query="dictId" type="number[]">
  Filter by one or more specific dictionary IDs.
</ParamField>

<ParamField query="status" type="number">
  Exact filter on status (`1` = active, `0` = disabled).
</ParamField>

<ParamField query="pageSize" type="number" default="10">
  Records per page.
</ParamField>

<ParamField query="pageIndex" type="number" default="1">
  Page number (1-indexed).
</ParamField>

#### Response fields

<ResponseField name="count" type="number">
  Total matching records.
</ResponseField>

<ResponseField name="data" type="object[]">
  <Expandable title="Dictionary type object fields">
    <ResponseField name="id" type="number">Unique dictionary type ID (stored as `dict_id`).</ResponseField>
    <ResponseField name="dictName" type="string">Human-readable name for the category (e.g., `"User gender"`).</ResponseField>
    <ResponseField name="dictType" type="string">Machine-readable key used to look up options (e.g., `"sys_user_sex"`).</ResponseField>
    <ResponseField name="status" type="number">Status (`1` = active, `0` = disabled).</ResponseField>
    <ResponseField name="remark" type="string">Optional description.</ResponseField>
  </Expandable>
</ResponseField>

<Tabs>

## Dictionary data endpoints

Dictionary data entries are the individual label/value pairs within a type. For example, the `sys_user_sex` type might contain entries `Male → 1`, `Female → 2`, `Unknown → 0`.

<Tip>
  Use `GET /api/v1/dict-data/option-select?dictType=sys_user_sex` to fetch a compact `label`/`value` list suitable for populating a dropdown — it returns only the two fields you need without pagination overhead.
</Tip>

### GET /api/v1/dict/data

Return a paginated list of dictionary data entries.

#### Query parameters

<ParamField query="dictType" type="string">
  Partial-match filter on the dictionary type key. Use this to retrieve all entries for one category.
</ParamField>

<ParamField query="dictLabel" type="string">
  Partial-match filter on the display label.
</ParamField>

<ParamField query="dictValue" type="string">

</ParamField>

<ParamField query="status" type="string">
  Exact filter on the entry status.
</ParamField>

<ParamField query="pageSize" type="number" default="10">
  Records per page.
</ParamField>

<ParamField query="pageIndex" type="number" default="1">
  Page number (1-indexed).
</ParamField>

#### Response fields

## Audit logs

go-admin records two types of audit logs: **operation logs** (one record per API request) and **login logs** (one record per authentication attempt). Both are read-only resources — records are created automatically by middleware and can only be read or deleted.

### GET /api/v1/sys-opera-log

Return a paginated list of operation log entries. All entries are automatically created by the logger middleware when `enableddb` is `true` in `settings.yml`.

**Query parameters**

<ParamField query="title" type="string">Partial-match filter on the operation module title.</ParamField>
<ParamField query="method" type="string">Partial-match filter on the handler function name.</ParamField>
<ParamField query="requestMethod" type="string">Exact filter on the HTTP method (e.g. `GET`, `POST`).</ParamField>
<ParamField query="operUrl" type="string">Partial-match filter on the request URL.</ParamField>
<ParamField query="operIp" type="string">Partial-match filter on the client IP address.</ParamField>
<ParamField query="status" type="string">Exact filter on status: `"1"` = success, `"2"` = error.</ParamField>
<ParamField query="beginTime" type="string">Filter records on or after this time (format: `2006-01-02 15:04:05`).</ParamField>
<ParamField query="endTime" type="string">Filter records on or before this time.</ParamField>
<ParamField query="pageSize" type="integer" default="10">Records per page.</ParamField>
<ParamField query="pageIndex" type="integer" default="1">Page number (1-indexed).</ParamField>

**Response fields**

<ResponseField name="data" type="object[]">

    <ResponseField name="deptName" type="string">Department name of the operator.</ResponseField>
